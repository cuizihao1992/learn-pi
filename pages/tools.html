Part 4

工具调用系统：模型怎么“操作电脑”

模型本身不能读文件、改文件、跑命令。它只能提出 toolCall。真正执行的是 Pi Agent。

工具调用生命周期

assistant message contains toolCall
  -> executeToolCalls()
  -> prepareToolCall()
  -> validateToolArguments()
  -> beforeToolCall hook
  -> tool.execute()
  -> afterToolCall hook
  -> createToolResultMessage()
  -> push toolResult back to context

为什么有 ToolDefinition 和 AgentTool 两层

ToolDefinition

面向描述、扩展系统、UI 渲染和工具管理。它告诉系统这个工具叫什么、参数是什么、如何展示。

AgentTool

面向 agent loop 执行。它包含 schema 和 execute 函数，是模型 toolCall 最终落地的地方。

内置工具

  • `read`：读取文件内容，通常是模型理解项目的第一步。
  • `bash`：执行命令，验证、搜索、构建、测试。
  • `edit`：按 oldText/newText 修改文件，并生成 diff。
  • `write`：写入新文件或覆盖文件。
  • `grep/find/ls`：只读探索工具，用于定位文件和内容。

edit 工具为什么复杂

编辑文件不是简单字符串替换。它要防止替错位置、处理不同换行符、避免并发写同一文件、生成可读 diff，并且在 oldText 不精确时给出可理解错误。

  • `edit.ts`：参数准备、路径解析、工具定义、执行。
  • `edit-diff.ts`：内容匹配、diff 生成、换行符处理。
  • `file-mutation-queue.ts`：同一路径写操作串行化。

为什么工具结果还要发回模型

因为 Pi Agent 不是预先写死流程。模型读完 README 后才知道要怎么编辑；编辑后看到 diff 才决定是否测试；测试后看到错误才决定是否修复。每个 toolResult 都是下一轮推理的输入。
