Part 8

为什么这样设计？还有其他方案吗？

理解架构不能只背模块名，还要知道为什么不用别的方案。下面是 Pi Agent 的几个关键取舍。

方案一：单体大循环

把 CLI 输入、模型请求、工具执行、UI 输出全写进一个大 while loop。

  • 优点：一开始写起来快。
  • 缺点：难测试、难扩展、难复用、provider 差异会污染主流程。
  • Pi 的选择：把主循环放到 agent-core，把产品能力放到 coding-agent。

方案二：模型直接控制所有流程

让模型决定什么时候读写、怎么存会话、怎么更新 UI。

  • 优点：看起来智能。
  • 缺点：不可控，安全边界和状态一致性差。
  • Pi 的选择：模型只产生 assistant message 和 toolCall，真实执行由工具系统控制。

方案三：每个 provider 单独写一套 Agent

  • 优点：能用 provider 特性。
  • 缺点：重复代码巨大，行为难保持一致。
  • Pi 的选择：pi-ai 统一 provider，Agent loop 面向统一 Message/EventStream。

方案四：不做事件流，最后一次性返回

  • 优点：实现简单。
  • 缺点：用户看不到进度，工具执行不可观察，扩展也难插入。
  • Pi 的选择：事件驱动，前端、会话保存、扩展都订阅事件。

方案五：永远保留完整历史

  • 优点：不会丢信息。
  • 缺点：很快超过模型上下文，成本高。
  • Pi 的选择：保留近期消息，把旧历史压缩成 summary。

设计原则总结

模型负责推理
工具负责执行
Agent loop 负责调度
AgentSession 负责产品运行时
pi-ai 负责 provider 差异
事件负责观察和前端更新
session/compaction 负责长期记忆
