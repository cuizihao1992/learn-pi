General Charter

源码学习实验室总宪章

这个网站不是文章堆叠，而是一个可长期扩展的学习系统。总宪章规定所有模块共同遵守的内容标准；每个模块再有自己的模块宪章，规定该专题的学习目标、边界、源码入口和扩展计划。

一、总目标

帮助新手从“会用 API”走到“看懂系统怎么设计、怎么运行、怎么调试、怎么扩展”。每个专题都必须围绕完整链路，而不是只列概念。

二、内容必须回答的 8 个问题

  1. 它解决什么问题？先讲场景，再讲对象名。
  2. 最小可运行例子是什么？用小代码建立直觉。
  3. 完整运行链路是什么？从用户 API 到底层执行逐步展开。
  4. 核心抽象有哪些？解释边界、职责和关系。
  5. 为什么这样设计？必须对比替代方案。
  6. 怎么调试？给故障表、检查顺序、关键状态。
  7. 源码从哪里读？给文件路径、函数入口、调用图。
  8. 怎么动手做？给练习任务和完成标准。

三、推荐模块目录结构

你的判断是合理的：每个模块放在独立目录下更适合长期扩展。当前固定为五大块：Pi Agent、Mini Agent、Cesium、OpenLayers、Gaussian Splatting。这样每个专题的宪章、页面、示例、资源互不污染，后续加 React、Mapbox、LangChain、数据库内核等专题也不会乱。

modules/{module-slug}/
  charter.html          # 模块宪章：目标、边界、章节标准
  index.html            # 模块入口
  lifecycle.html        # 完整运行链路
  internals.html        # 核心机制
  debugging.html        # 调试手册
  glossary.html         # 术语表
  labs.html             # 动手练习
  assets/               # 图片、图表、示例数据
examples/{module-slug}/
  ...                   # 可运行示例代码

四、模块化后的兼容策略

专题内容已经整理到 `modules/` 目录下；旧的公开 URL，例如 `pages/mini-agent.html`、`pages/cesium-render-path.html`，保留为兼容跳转页，避免旧链接失效。后续策略是：

  • `pages/` 只放全站级页面和旧链接跳转页。
  • `modules/` 是未来内容规范目录。
  • 每个模块先补模块宪章，后续新增页面优先放入对应模块目录。
  • 必要时再逐步加 redirect/索引页，而不是一次性迁移。

五、模块宪章必须包含什么

  • 学习对象：这个模块研究哪个项目/系统。
  • 目标读者：小白、进阶、源码读者、工程实践者。
  • 完成标准：学完能说清楚什么、能实现什么。
  • 核心链路：一条最重要的数据流/控制流。
  • 源码入口：优先阅读的文件和函数。
  • 调试入口：常见问题从哪里看。
  • 未来扩展：后续可以继续补哪些章节。

六、写作原则

  • 默认读者是新手，但不降低工程深度。
  • 每个概念都要能落到源码文件。
  • 每条链路都要说明“谁产生、谁消费、为什么需要”。
  • 代码块要能复制，示例要尽量可运行。
  • 既讲实现，也讲取舍。
  • 每个模块都要有调试手册和练习任务。
