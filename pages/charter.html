源码学习实验室内容宪章

这个网站后续会不断加入新项目。为了不变成零散笔记，每个新专题都按同一套结构写：先让小白知道它解决什么问题，再讲完整链路，再落到源码和练习。

每个专题必须回答的 8 个问题

  1. 它解决什么问题？先讲使用场景，不先堆类名。
  2. 最小可运行例子是什么？用一段代码建立直觉。
  3. 完整运行链路是什么？从用户 API 到底层执行，一步一步画出来。
  4. 核心抽象有哪些？解释对象之间的边界和关系。
  5. 为什么这样设计？必须和其他方案对比。
  6. 怎么调试？列出常见故障、检查顺序、关键状态。
  7. 源码从哪里读？给出文件路径、函数入口和调用图。
  8. 怎么动手做？给出练习任务和完成标准。

统一章节模板

专题总览
  -> 最小例子
  -> 完整运行路径
  -> 核心概念和术语
  -> 关键源码入口
  -> 为什么这样设计
  -> 调试手册
  -> 动手练习
  -> FAQ / 常见误区

写作原则

  • 从问题开始：先说“为什么需要这个对象”，再说对象叫什么。
  • 坚持完整链路：不要只讲 API，要讲数据如何一路流到执行层。
  • 代码和概念互相映射：每个抽象都要能指到源码文件。
  • 给替代方案：讲“为什么不是另一种设计”。
  • 默认读者是新手：避免“显然”“很简单”这种跳步。
  • 保留工程判断：说明性能、复杂度、可测试性和扩展性取舍。
