Part 2

完整例子：从一句话到模型、工具、前端

假设用户输入：帮我在 README 里增加安装说明，然后运行测试确认没有问题。下面按真实运行顺序拆开。

0. 原始输入

帮我在 README 里增加安装说明，然后运行测试确认没有问题

这句话先进入某个模式层：交互终端是 `interactive-mode.ts`，一次性输出是 `print-mode.ts`，机器调用是 `rpc-mode.ts`。这些模式层都不会自己实现 agent 推理，而是调用 `AgentSession.prompt()`。

1. AgentSession.prompt 做预处理

  • 判断当前 agent 是否正在运行。如果正在运行，新输入可能进入 steering 或 follow-up 队列。
  • 展开 prompt template，例如用户输入某个 slash command 或模板命令时，会把参数替换进模板。
  • 合并图片附件，形成文本 + image content。
  • 加入 pending next-turn messages，例如扩展或系统临时追加的上下文。
  • 构造 `role: "user"` 的 AgentMessage。
{
  role: "user",
  content: "帮我在 README 里增加安装说明，然后运行测试确认没有问题",
  images: [],
  source: "interactive"
}

2. Agent.run 进入核心循环

`Agent` 会把当前状态转换成 `AgentLoopConfig`。这里包含模型、工具列表、系统提示词、streamFn、beforeToolCall、afterToolCall、prepareNextTurn、队列读取函数等。

Agent.state
  messages: 历史消息 + 当前用户消息
  tools: read, bash, edit, write, grep, find, ls
  model: 当前选择的大模型
  thinkingLevel: off/minimal/low/medium/high

3. runLoop 第一次调用模型

模型看到的不是一句话，而是完整上下文。

[
  system: "你是 Pi coding agent... 工具使用规则... 项目上下文...",
  user/assistant/toolResult: "历史消息，或压缩后的 summary",
  user: "帮我在 README 里增加安装说明，然后运行测试确认没有问题"
]

tools = [read, bash, edit, write, grep, find, ls]

4. 模型决定先读文件

模型不能直接访问文件系统，它只能返回 toolCall。

{
  type: "toolCall",
  id: "call_read_1",
  name: "read",
  input: { "path": "README.md" }
}

`runLoop` 发现 assistant message 里有 toolCall，于是进入 `executeToolCalls()`。

5. Pi 执行 read 工具并回填结果

prepareToolCall
  -> 找到 read 工具
  -> validateToolArguments 校验参数
  -> beforeToolCall hook
  -> executePreparedToolCall
  -> afterToolCall hook
  -> createToolResultMessage
{
  role: "toolResult",
  toolCallId: "call_read_1",
  content: "README.md 当前内容..."
}

6. 第二次调用模型：模型基于文件内容决定编辑

{
  type: "toolCall",
  id: "call_edit_1",
  name: "edit",
  input: {
    "path": "README.md",
    "edits": [
      {
        "oldText": "## 本地预览\n\n直接用浏览器打开 `index.html` 即可。",
        "newText": "## 安装\n\n```bash\nnpm install\n```\n\n## 本地预览\n\n直接用浏览器打开 `index.html` 即可。"
      }
    ]
  }
}

7. edit 工具不是简单替换字符串

edit 工具会处理路径解析、换行符、BOM、oldText 唯一性、模糊匹配、diff 生成、并发写入队列等问题。执行后会把修改结果和 diff 返回给模型。

8. 模型继续调用 bash 验证

{
  type: "toolCall",
  id: "call_bash_1",
  name: "bash",
  input: { "command": "npm test", "timeout": 120000 }
}

bash 输出会被截断、清理二进制内容，并作为 toolResult 回到上下文。

9. 模型生成最终回答

当模型不再返回 toolCall，而是返回普通文本，runLoop 结束当前 turn。如果没有 follow-up message，agent_end 发出，本次运行结束。

已完成：
- README.md 已增加安装说明
- 已运行 npm test
- 测试通过

整个过程都在发事件，UI 不是最后才知道结果。

tool_execution_end(read)
