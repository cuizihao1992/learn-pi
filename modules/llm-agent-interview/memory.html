Memory

长期陪伴型 AI 角色的记忆机制怎么设计

核心难点是：对话轮次增多后，不能把全量历史都塞进 prompt。这样会成本高、延迟高、注意力分散，还会让模型在大量无关历史里丢失重点。面试答题要围绕“分层存储、按需召回、动态更新、冲突清理、隐私权限”展开。

一、三类记忆

记忆类型存什么怎么存怎么召回
短期记忆最近几轮原始对话、当前任务状态会话内 messages / rolling window直接放入 prompt，保证上下文连贯。
实体画像记忆用户偏好、身份、经历、关系、长期事实结构化表、KV、JSON profile按 user_id 精准读取，可按字段过滤。
长期情景记忆感悟、闲聊、重要片段、事件经过切片后 embedding，存向量库按语义相似召回，再过滤和重排。

二、为什么要分层

事实型信息适合结构化存储，因为它需要可覆盖、可校验、可冲突处理。例如“用户喜欢日式装修”可以放在 profile 里。情景型信息适合向量存储，因为它不一定有固定字段，例如“一次关于职业焦虑的长对话”更适合按语义召回。最近对话则必须原样保留，因为它承载代词、指令和当前任务状态。

三、调用链路

用户输入
  -> 读取短期对话窗口
  -> 读取用户画像 profile
  -> 用 query 检索长期情景记忆
  -> 过滤权限、时间、置信度
  -> rerank 选出少量高价值片段
  -> 拼接 system prompt + profile + memories + recent messages
  -> 调用大模型
  -> 返回用户
  -> 后台异步抽取新记忆

四、后台异步记忆更新

记忆抽取不要阻塞用户响应。更好的做法是主链路先回答，后台任务再处理记忆。

对话结束事件
  -> memory extraction job
  -> 抽取事实、偏好、事件、情绪标签
  -> 判断是否值得写入
  -> 与旧记忆做冲突检测
  -> 更新 profile 或写入向量库
  -> 记录来源、时间、置信度
  -> 周期性遗忘和压缩

五、冲突与遗忘机制

  • 冲突：旧记忆说用户喜欢咖啡，新对话说最近不喝咖啡。不要简单追加，要标记时间和状态变化。
  • 时效：短期偏好容易过期，长期事实不应随便覆盖。
  • 置信度：“我可能喜欢”不如“我最喜欢”置信度高。
  • 遗忘：低价值、低置信、长期未召回的情景记忆可以压缩或删除。
  • 隐私：敏感记忆要有权限、可见性、删除能力。

六、面试答题模板

我会把记忆分为短期上下文、结构化用户画像和长期情景记忆。
短期上下文直接保留最近 N 轮原始对话。
稳定事实进入 profile，用字段化结构支持精准检索和冲突覆盖。
非结构化对话片段做 chunk + embedding，进入向量库，按语义召回。
在线请求时拼 system prompt、profile、召回片段和近期对话。
记忆写入采用异步任务，做抽取、去重、冲突处理、置信度和遗忘。
这样可以兼顾成本、延迟、准确性和长期陪伴体验。
