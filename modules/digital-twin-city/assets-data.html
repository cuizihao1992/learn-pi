Assets and Data

三维资产和城市数据如何组织

数字孪生的资产不是只有一个三维模型。它通常由基础地理数据、建筑模型、BIM、倾斜摄影、点云、实时传感器、业务系统数据和历史数据共同组成。难点在于坐标统一、LOD、对象 ID、数据更新和权限。

资产分类

类型常见格式用途注意点
二维基础数据GeoJSON、MVT、Shapefile、PostGIS道路、地块、行政区、管线轮廓投影、拓扑、属性字段。
建筑白模3D Tiles、glTF城市体块、楼栋定位轻量、ID 绑定、按区块切片。
BIMIFC、Revit、glTF、3D Tiles建筑内部构件、机电设备模型太重，需要简化和权限控制。
倾斜摄影3D Tiles、OSGB 转换大范围真实城市外观数据量大，纹理和 LOD 重要。
点云LAS/LAZ、3D Tiles Point Cloud测绘、道路、树木、电力巡检点数巨大，需要抽稀和分级。
高斯泼溅ply/splat/自定义格式局部真实感重建浏览器渲染和地理配准仍需工程处理。
实时数据MQTT、Kafka、WebSocket、REST车辆、传感器、告警、轨迹延迟、乱序、去重、历史归档。

数据治理链路

原始数据
  -> 坐标转换
  -> 清洗和校验
  -> 对象 ID 绑定
  -> 切片 / LOD / 压缩
  -> 元数据入库
  -> 发布为 tile/API
  -> 前端按需加载

对象 ID 比模型更重要

如果没有统一 ID，三维模型只是好看的壳。系统必须能从用户点击的建筑跳到属性、工单、设备、告警、历史曲线和 Agent 分析。ID 设计要稳定，不能依赖临时文件名或前端数组下标。

更新策略

  • 基础地图：低频更新，按版本发布。
  • 建筑模型：中低频更新，按区域切片，支持灰度替换。
  • 传感器数据：实时入库，前端订阅最新状态。
  • 轨迹数据：实时展示最新点，历史轨迹进时序库或轨迹库。
  • 告警事件：必须可追溯，有状态流转和处置记录。
