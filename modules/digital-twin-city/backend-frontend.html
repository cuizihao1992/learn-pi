Architecture

数字孪生城市前后端架构

一个可维护的数字孪生系统，需要把瓦片服务、业务 API、实时推送、Agent 工具、权限审计和前端状态管理拆清楚。不要把所有逻辑塞进一个大屏页面。

后端服务分层

API Gateway
  -> Auth / RBAC / Audit
  -> Map Tile Service
  -> 3D Tiles Service
  -> Spatial Query Service
  -> Realtime Event Service
  -> Time Series Service
  -> Asset Metadata Service
  -> Agent Tool Service
  -> Report / Export Service

数据库选择

数据推荐存储原因
空间对象PostgreSQL + PostGIS空间索引、缓冲区、相交、距离查询成熟。
时序传感器TimescaleDB / InfluxDB / ClickHouse按时间范围聚合和降采样。
事件告警PostgreSQL / Elasticsearch / OpenSearch状态流转、全文搜索、时间筛选。
大模型和报告对象存储 + 关系库元数据报告文件、截图、附件和审计记录。
瓦片和三维资产对象存储 / CDN静态资源大、适合缓存。

前端结构

App Shell
  -> Map View: Mapbox / Cesium / OpenLayers
  -> Layer Tree
  -> Object Inspector
  -> Timeline
  -> Alert Panel
  -> Search Panel
  -> Agent Chat Panel
  -> Charts and Reports

前端状态要分层：地图相机状态、图层可见性、选中对象、时间范围、订阅状态、Agent 对话状态。地图对象很多时，不要把每个要素都放进前端全局状态；应让地图引擎和瓦片系统管理渲染对象，只把业务选择结果放入状态。

权限与审计

数字孪生涉及城市数据和设备控制，权限必须严肃处理。不同角色看到的图层、对象属性、摄像头、控制按钮都应不同。Agent 的每次工具调用也应该被记录：谁问的、调用了哪个工具、参数是什么、返回了什么、是否触发外部动作。
