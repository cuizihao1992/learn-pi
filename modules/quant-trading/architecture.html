Architecture

一个量化交易系统通常长什么样

量化系统可以很小，也可以很复杂。入门项目可能只是一个 Python 脚本；机构系统会拆成数据服务、研究平台、回测引擎、订单管理、风控、监控和审计。关键是每一层边界清晰。

核心模块

Data Ingestion
  -> Data Store
  -> Research Notebook
  -> Feature / Signal Engine
  -> Backtest Engine
  -> Portfolio / Risk Engine
  -> Execution Engine
  -> Broker / Exchange API
  -> Monitor / Alert / Log

1. 数据层

数据层负责把外部行情、财务、公司行动、指数成分、交易日历变成可查询、可复现的数据。它要处理复权、时区、缺失值、异常价格、停牌、拆股、分红。量化里最贵的错误常常不是模型错，而是数据错。

  • 行情数据：日线、分钟线、tick、盘口、成交明细。
  • 基本面数据：财报、估值、盈利预测、行业分类。
  • 另类数据：新闻、舆情、卫星图像、供应链、网页数据。
  • 元数据：交易日历、复权因子、成分股历史、停牌和涨跌停。

2. 研究层

研究层通常用 notebook 或脚本探索因子和策略。好的研究从假设开始，而不是从暴力搜索参数开始。例如：“过去 12 个月强、过去 1 个月不过热的股票，未来一个月是否有动量延续？”这个假设可以被数据检验。

3. 回测层

回测层要像历史真实发生一样推进时间：在某个时间点只允许使用当时已经知道的数据，然后生成订单，再按成交规则撮合。事件驱动回测系统会按数据事件、信号事件、订单事件、成交事件推进，Zipline 这类系统就是典型的事件驱动思路。

回测要素为什么重要
手续费高换手策略收益会被交易成本吃掉。
滑点回测价格和真实成交价格不可能完全一样。
冲击成本大资金下单会影响价格，小资金可忽略但也要理解。
成交限制不能假设在涨停、停牌、低流动性时无限成交。
延迟信号生成、下单、成交之间有时间差。

4. 组合和风控层

信号只告诉你买什么，组合层决定买多少。它要考虑单票上限、行业暴露、杠杆、现金、最大回撤、波动率目标、相关性、止损和再平衡。很多策略单看信号不错，但组合后风险集中，实盘很容易出问题。

5. 执行层

执行层负责把目标仓位变成真实订单。它要处理订单拆分、限价单、市价单、撤单、重试、成交回报、券商 API 断线、重复下单保护。实盘里最重要的不是聪明，而是不能乱下单。

6. 监控和审计

任何自动交易系统都必须有监控。至少要监控行情延迟、账户权益、持仓、未成交订单、异常成交、策略进程、服务器资源、网络连接、日志错误。出现异常时，应能自动停止交易或进入只减仓模式。
